# IndustryPulse 产品与架构分析

## 一、产品定位

产业热点事件时间轴工具——将跨来源的产业资讯聚合为带有时间脉络的事件流，并借助 AI 自动生成事件解读，帮助用户快速建立对产业趋势的认知。

核心场景：分析师写报告、投资经理做调研、战略部看竞品、记者找选题。

---

## 二、目标用户

| 用户角色 | 核心需求 |
|----------|----------|
| 产业分析师 | 按时间线梳理事件，形成报告素材 |
| 投资经理 | 追踪被投企业及行业动态，辅助决策 |
| 企业战略部 | 监控竞品和行业热点 |
| 记者/编辑 | 发现选题线索，追踪事件进展 |
| 学术研究者 | 收集产业演变数据，支撑学术分析 |

---

## 三、核心价值

- **效率**：数天的信息收集压缩到分钟级
- **洞察**：AI 自动提炼影响，无需逐篇阅读原文
- **脉络**：时间轴呈现，清晰展示事件演进

---

## 四、功能架构

### 4.1 五大核心功能

| 功能 | 说明 |
|------|------|
| **时间轴搜索** | 关键词 + 时间范围双维度筛选，生成结构化时间线 |
| **热点追踪** | 抓取百度热搜，实时反映短期热度 |
| **订阅源管理** | 可增删、启用/禁用新闻来源，支持多类型（官方/媒体/学术/社交/数据） |
| **收藏与标注** | 用户收藏事件并添加个人注解 |
| **数据分析** | 事件类型统计、热度趋势、分布图表 |

### 4.2 数据管道

```
订阅源（RSS/HTML/Sitemap）
       ↓ 定时爬取（30分钟）
  关键词匹配 → 生成AI解读 → 写入events表
       ↓
  用户按需查询 → 时间轴展示
```

---

## 五、数据模型设计

| 表 | 职责 | 设计思路 |
|----|------|----------|
| `events` | 事件主体 | 核心数据，存标题/摘要/AI解读/来源/发布时间 |
| `industries` | 产业赛道 | 预置8个行业 + 用户可添加，关键词驱动爬取 |
| `sources` | 订阅源 | 含crawl_type决定爬取方式，含selector配置 |
| `favorites` | 收藏 | 关联用户+事件，支持annotation注解 |
| `users` | 用户 | 简化模型 |

关键决策：MySQL 而非 MongoDB，因为industries和sources与events存在外键关联，关系型更清晰。

## 六、订阅源分类体系

18个预置来源分为五类，分类决定了爬取策略：

| 类型 | 说明 | 爬取方式 |
|------|------|----------|

| data | 数据平台 | RSS |

## 七、优点

### 1. 产品解决了真实痛点
信息过载是产业分析领域的核心问题，该工具通过自动聚合 + AI 解读，大幅降低了信息获取成本。

### 2. 多维度筛选设计合理
关键词 + 时间范围的双维度设计，覆盖了"我要看某行业最近发生了什么"和"我要看某个时间段的全部动态"两种核心需求。

### 3. 订阅源可动态管理
用户可以随时启用/禁用、增删订阅源，数据管道是数据驱动的（从DB读取关键词和来源），而非硬编码。

### 4. AI 解读提升信息密度
不仅聚合内容，还通过 AI 生成事件影响点评，将原始咨询升级为可操作的洞察。

### 5. 事件类型分类有意义
policy/funding/product/ma/tech/report/person 等标签，帮助用户快速定位感兴趣的事件类型。

### 6. 导出功能覆盖核心场景
支持 JSON 和 Markdown，满足报告撰写和数据二次处理的需求。

---

## 八、缺点与风险

### 1. 搜索体验依赖关键词质量
目前industry表里的关键词是预设的，用户无法直接输入自由搜索Query，只能在预设赛道中选择。如果用户的关注点不在预设范围内，体验受限。

### 2. "热点"功能过于简单
热搜直接依赖 SerpAPI，没有对来源和可信度的过滤，噪音较大，且与主时间轴功能割裂。

### 3. 用户体系薄弱
users表极为简化，目前只有weiyu一个用户，无登录机制，无权限设计。favorites功能看似支持多用户，实际单用户。无法扩展为团队协作场景。

### 4. 缺乏事件关联性设计
事件之间没有关联/引用/延伸关系，用户看到的是一个一个孤立的事件点，无法追溯一个事件的完整生命周期（前因、后续、相关方）。

### 5. 时间轴的叙事能力不足
时间轴只是按时间排序的事件列表，没有里程碑标注、阶段划分、因果连线等叙事辅助，信息密度有待提升。

### 6. 数据 freshness 不确定性
爬虫依赖来源的RSS/HTML稳定性，但没有监控告警机制，一旦来源失效或格式变化，用户无感知。

### 7. 缺少内容去重和可信度评估
同一事件可能出现在多个来源，时间轴没有做去重或可信度加权，用户可能看到重复内容。

### 8. 国际化扩展未考虑
所有文案、预设来源、关键词均为中文，架构层面没有为多语言场景预留设计。

---

## 九、架构层面的关键决策回顾

| 决策 | 选择 | 理由 |
|------|------|------|
| 后端语言 | Python FastAPI | AI服务集成方便，异步支持好 |
| 数据库 | MySQL | 结构关系清晰，外键关联多 |
| 缓存层 | 未引入 | Demo阶段数据量不需要 |
| 爬取策略 | 定时批量爬取 | 保证数据新鲜度，用户无需主动触发 |
| AI Commentary | 规则降级 | 无Key时仍能提供基础解读 |

---

## 十、总结

| 维度 | 评价 |
|------|------|
| 产品定位 | 清晰，瞄准产业信息聚合的真实需求 |
| 功能完整性 | 核心链路完整（爬→存→查→AI→展示），但关键场景（如搜索精度、内容去重、事件关联）有缺失 |
| 数据模型 | 合理，但用户体系和事件关联性薄弱 |
| 扩展性 | 订阅源可动态管理，但缺乏事件关系图谱的扩展基础 |
| 演进方向 | 目前是"信息聚合工具"，潜力方向是"产业情报平台"——这需要内容关联、团队协作、数据订阅等能力的支撑 |
